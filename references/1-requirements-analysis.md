# 需求分析模块

## 目标

把模糊需求转成有范围、有业务规则、可验收的用例契约。本模块是每次执行开发罗盘的第一个必选阶段；完成后继续形成开发思路，不得只在内部分析而不交付结果。

## 1. 建立需求基线

先检查用户描述、现有代码、测试、接口、数据结构和文档。输出：

- 核心价值：用户或业务最终获得什么可观察结果；
- 本次范围：必须做、可后置、明确不做；
- 事实与证据：已存在的行为和约束；
- 未知项：阻塞决策、安全假设、非阻塞问题；
- 术语表：同一概念统一命名，标出歧义词。

## 2. 使用“角色—信息—动作”拆解

### 角色

列出人、外部系统、定时任务等发起者。每个角色记录身份、职责、可见范围、所有权、权限边界及角色继承关系。

### 信息

先把信息视为业务概念，不要过早等同数据库表。每项记录：

- 业务含义与权威所有者；
- 与其他信息的关系和基数；
- 生命周期、关键状态和不变量；
- 敏感字段、外部契约或合规要求。

### 动作

CRUD 只描述信息维护。核心业务使用领域动词，如提交、审批、下单、支付、发货、退款。每个动作补全前置条件、权限、状态变化、副作用、失败结果、幂等和并发规则。

## 3. 编写用例

使用句式：

> 谁，在什么前置条件下，对什么信息执行什么业务动作，触发哪些状态或副作用，得到什么可观察结果。

| 字段 | 必填内容 |
| --- | --- |
| UC | 唯一编号、业务目标 |
| 角色 | 发起者、权限和所有权范围 |
| 前置条件 | 当前状态、认证、依赖条件 |
| 信息对象 | 操作对象及其权威来源 |
| 动作 | 领域动词 |
| 状态与副作用 | 状态迁移、事件、通知、持久化等 |
| 可观察结果 | 返回值、页面、事件、数据变化 |
| 异常与边界 | 越权、非法状态、外部失败、重复请求 |

至少覆盖成功路径、未认证、无权限、非法输入、相反状态、不存在资源、重复执行和关键副作用不应发生的路径。

## 4. 转成验收用例

为范围内每个用例写至少一个 Given/When/Then：

```text
Given 角色、初始状态、必要数据和外部条件
When  执行一个明确业务动作
Then  验证可观察结果、状态变化和必要副作用
And   验证禁止发生的变化（适用时）
```

验收标准必须二元可判定。避免“功能正常”“性能良好”等不可验证描述；性能要求应给出数据规模、指标和阈值。

## 5. 补齐横切约束

按风险选择：安全与权限、幂等、并发、事务、一致性、审计、可观测性、兼容性、迁移、回滚、性能、隐私和无障碍。

## 门禁

只有同时满足以下条件才算完成：

- 核心目标和范围可观察、可验证；
- 每个动作有授权角色，每项核心信息有所有者；
- 每个用例有前置、结果、异常和至少一个验收用例；
- 状态迁移和跨角色交接可追踪；
- 事实、推断和未知项已分开；
- 阻塞性业务决策已明确提出。

## 输出

在最终答复中使用固定 Markdown 二级标题 `## 需求分析`。依次输出：需求基线、角色表、信息模型、用例表、Given/When/Then、横切约束、范围清单、未知项与门禁结论。

产物按任务复杂度缩放：简单任务可以合并表格并保持简短，但至少明确核心目标、范围、关键验收行为、约束和未知项。已有需求文档可以复用，但必须摘要本次采用的结论，不能只给链接或声称“已分析”。
