## 目标与工作合同

把一个可观察行为通过有效红灯、最小绿灯和安全重构实现。Bug 修复先留下能稳定复现缺陷的永久回归测试。

开始前读取现有测试、测试配置、工厂、公共夹具和项目命令，沿用目标项目的语言、框架、命名和目录约定。先写出一句行为合同：

```text
角色/主体 + 关键初始状态 + 动作 + 可观察结果 + 不应发生的变化
```

不要为方法数量、覆盖率数字或目录对称性机械增加测试。只测试公共行为、业务规则、边界契约和真实风险。

## 1. 建立绿灯基线与最小切片

先运行现有相关测试。若基线失败，记录命令和失败摘要，区分旧问题与本次变更；不要把旧失败当作新红灯。

从验收用例选择一个最小纵向切片。优先交付一个真实闭环，而不是先把所有类、表和接口搭完。一次循环只推进一个行为：

```text
行为合同 → 聚焦测试 → 有效 RED → 最小 GREEN → REFACTOR → 扩大验证
```

## 2. 选择测试层级

选择足以证明风险的最低层级，不追求固定比例，也不按目录名称判断测试类型。

| 层级 | 验证对象 | 边界 |
| --- | --- | --- |
| 领域单元测试 | 纯计算、VO、Entity、不变量、状态迁移 | 不启动框架，不访问数据库、网络或文件系统 |
| 应用用例测试 | 一个命令或用例的编排、权限与副作用决策 | 使用内存替身或受控依赖 |
| 组件/集成测试 | ORM、Repository、迁移约束、序列化、适配器 | 跨越一个真实技术边界 |
| HTTP/功能测试 | 路由、中间件、验证、授权、响应及最终状态 | 启动应用并可访问测试数据库 |
| 契约测试 | 多个实现是否遵守同一端口或共享行为 | 同一测试集运行于每个实现 |
| 端到端测试 | 浏览器、客户端脚本、会话和关键用户旅程 | 完整系统，数量少、环境独立 |

使用数据库、ORM 或数据库重置夹具的测试属于集成测试，即使文件被放在“单元测试”目录。人工浏览器检查适合探索和视觉确认，不能替代可重复的行为回归。

### 外向内推进

复杂功能优先从最小 HTTP/用例测试定义可观察结果。当外层测试暴露出独立业务能力时：

1. 暂停外层测试；
2. 用更窄的领域、组件或适配器测试驱动该能力；
3. 让窄测试转绿；
4. 回到原外层测试完成闭环；
5. 保留各层不同的契约，删除重复验证。

内部辅助方法若只服务于已覆盖的公共行为，不单独测试。新方法承载业务规则、跨入口复用或独立边界时才下沉测试。

## 3. 测试命名

让测试名像一条可执行规格，回答“谁，在什么条件下，做什么，结果如何”。语义优先于框架语法；使用项目已有的测试属性、`test*`、snake_case、自然语言 DSL 或其他约定。

### 命名公式

按场景选用一种稳定结构：

```text
<actor>_<expected_behavior>_when_<condition>
<subject>_<expected_outcome>_after_<action>
<action>_is_idempotent_for_<business_identity>
<query>_returns_<result>_when_<condition>
<field>_<rule>_when_<action>
```

例如：

```text
guest_is_redirected_to_login_when_posting_an_answer
question_owner_can_publish_a_draft
submitting_the_same_order_request_twice_creates_one_order
published_questions_exclude_drafts
content_is_required_when_posting_an_answer
subscriber_is_notified_when_another_user_answers
```

遵守以下规则：

- 使用统一语言和业务状态，如 `draft`、`published`、`verified`、`owner`、`stranger`；
- 一个名称表达一个规则；合并两个独立动作的测试要拆分或参数化；
- 负向名称明确被拒绝的角色、动作及禁止结果，统一项目中的 `cannot`/`is_forbidden` 风格；
- 幂等测试写明业务身份和最终基数，不只写 `works_only_once`；
- Bug 回归名称写出触发条件和错误结果，不使用工单号代替行为描述；
- 避免 `works`、`can_do_it`、`can_know_it`、`test_success`、`happy_path` 等模糊名称；
- 避免把控制器方法、私有方法、Trait 或 ORM 实现细节写入行为测试名；
- 让成对场景对称，如 owner/non-owner、authenticated/guest、valid/invalid。

### 类、文件、变量与辅助方法

- 功能测试类按能力命名，如 `PostAnswerTest`、`DeleteAnswerTest`、`FilterQuestionsTest`；
- 领域测试类按被测概念命名；共享行为使用 `...ContractTest`；
- 先按真实测试层级分目录，再按业务能力组织，不按控制器目录机械镜像；
- 变量使用角色和状态名，如 `owner`、`nonOwner`、`publishedQuestion`、`response`；避免 `data1`、`result`、`test`；
- 工厂状态使用领域词，如 `published()`、`unverified()`、`ownedBy()`，避免散落魔法字段；
- Arrange 辅助方法使用明确动词，如 `signInAs()`、`createPublishedQuestion()`；只隐藏机械样板，不隐藏关键前提、Act 或 Assert。

## 4. 测试结构、数据与断言

使用 Given/When/Then 或 Arrange/Act/Assert，并用空行分隔三个阶段。代码已表达意图时不要添加重复代码含义的编号注释。

```text
Arrange：准备最少的角色、状态、输入和依赖
Act：执行一个主要业务动作
Assert：验证主要结果、持久化状态、副作用和禁止变化
```

一个场景可以有多个断言；“一个行为”不等于“一个断言”。主要结果先断言，再验证持久化和副作用。幂等、状态机和重试场景可以有多个有意的动作，但要保持一个业务问题。

### 测试数据

- 让工厂默认生成完整有效对象，只在当前场景覆盖关键字段；
- 同时创建目标数据和相似但不应命中的干扰数据；
- 数据变更后重新加载实体或查询数据库，避免断言陈旧对象；
- 冻结时钟测试时间边界，覆盖过去、当前、未来及时区规则；
- 隔离数据库、缓存、文件、队列、邮件和会话，禁止读写开发或生产状态；
- 使用生产同类数据库验证锁、事务、唯一约束、JSON 和排序规则等数据库特性。

### 风险与边界

按需求选择成功、未认证、无权限、不存在、非法输入、相反状态、重复执行、并发、时间边界、外部失败和副作用不发生等场景。

验证范围或长度时优先覆盖等价类与边界：

```text
min - 1、min、max、max + 1，以及一个普通有效值
```

无效输入测试要保证其他字段有效，并断言准确错误字段及没有持久化或副作用。使用具名数据集表达每个变体，不在一个测试里写难以定位失败的循环。

### 断言合同

| 行为 | 至少验证 |
| --- | --- |
| 查询/页面 | 成功契约、准确成员与顺序、关键内容或视图数据 |
| 创建/更新 | 精确响应、业务标识、持久化状态、所有者及副作用 |
| 验证失败 | 指定字段/错误码，且无写入、无消息、无文件 |
| 游客操作 | 精确登录跳转或 401，且状态不变 |
| 已认证但无权限 | 403，且资源和副作用不变 |
| 删除 | 精确响应、目标消失、无关数据保留 |
| 关联 | 相关对象被包含、无关对象被排除、反向归属正确 |
| 分页/筛选 | 目标与干扰项、顺序、页大小、总数、筛选参数保留 |
| 异步 | 调度参数正确，以及任务执行后的最终结果 |

只断言 `200`、`302`、数量或内部方法调用通常过弱。断言最接近用户和业务的可观察结果；必要时再用交互断言证明边界调用。

### 异常处理

HTTP 测试默认保留框架异常处理，直接验证 302、401、403、404、409、422 等外部契约。仅在定位非预期 500 时临时暴露异常，找到原因后移除诊断开关并恢复行为断言。

异常本身就是当前层契约时，断言精确类型及必要的错误码或消息，并把异常期望紧邻 Act，避免 Arrange 阶段的同类异常误使测试通过。

## 5. 测试替身与技术边界

只替换进程外、慢、随机或难控制的边界，不替换本轮真正要验证的业务对象。

| 替身 | 用途 | 注意 |
| --- | --- | --- |
| Stub | 为依赖提供指定返回值 | 不用它证明交互发生 |
| Fake | 提供可工作的轻量实现 | 保持确定性，并按输入产生结果或记录输入 |
| Spy | 记录调用供事后检查 | 同时验证业务结果，避免只测调用次数 |
| Mock | 预先约束关键交互 | 只用于交互本身就是合同的边界 |

- 快速测试禁止真实访问网络、支付、邮件、对象存储和外部队列；
- Fake 不要对所有输入返回同一常量，否则会漏掉传错参数；
- 对真实适配器使用 HTTP Fake、组件测试或单独的契约/online 套件；
- 多个实现遵守同一端口时，让同一契约测试运行于每个实现；
- 至少出现两个真实使用者并稳定后再抽共享契约，避免猜测式抽象。

## 6. RED

运行新测试并保留第一次有效失败。有效 RED 必须满足：

- 基线为绿，或已明确记录旧失败；
- 测试确实被发现并可独立运行；
- 名称描述目标行为；
- 失败原因是目标行为缺失或错误，不是语法、环境、错误夹具、错误入口或过滤错测试；
- 断言足以证明业务缺陷，而非只证明路由、类、表或队列任务存在。

路由不存在、类不存在和迁移缺失可以是纵向搭建的中间红灯，但不能替代最终行为红灯。测试直接通过时检查：行为是否已经存在、断言是否过弱、夹具是否预先满足结果、测试是否未执行、是否只验证了调度而没有验证任务行为。

## 7. GREEN

只编写足以让当前失败通过的生产代码，每个小步重跑聚焦测试。错误从“类不存在”推进到“断言不符”是进展，但还不是绿灯。

绿灯门禁：

- 当前行为和相关模块测试通过；
- 拒绝路径没有非法写入或副作用；
- 异步行为分别验证“已调度”和“执行结果”；
- 外部失败和重试具有明确结果；
- 不能靠跳过测试、吞异常、放宽断言或修改需求换取通过。

Bug 修复只改能消除根因的最小边界；服务端规则是事实来源，隐藏页面按钮只能作为体验同步。

## 8. REFACTOR

只在绿色状态重构。清理重复、改善命名、移动职责、缩小夹具并抽取已出现的变化点。每个小改动后运行聚焦测试，结束后运行相关套件。

测试代码也需要重构，但遵守以下边界：

- 只抽取重复的 Arrange 样板，不隐藏关键场景差异；
- 共享行为确实适用于多个实现时抽契约测试；
- 保持测试名、Act 和主要 Assert 在文件中一眼可见；
- 不为追求 DRY 把测试变成带分支的第二套生产程序；
- 重构不得改变外部行为或削弱回归保护。

## 9. 专项行为矩阵

### 认证、授权与验证

- 覆盖游客、已认证非所有者、所有者，以及真实存在的管理员角色；
- 不信任请求中的用户或所有者标识，验证身份来自认证上下文；
- 授权发生在验证和写入之前；拒绝场景同时断言状态不变；
- 策略单元测试不能替代入口测试，入口测试证明路由、中间件和策略已经接线。

### 幂等与并发

- 验证首次执行、同一身份重复执行、不同身份执行和取消不存在关系；
- 断言最终只有一个业务结果且未抛未定义异常；
- 用数据库唯一约束和原子写入兜底；`exists()` 后 `create()` 不能证明并发安全；
- 高风险写入增加并发测试，验证冲突响应和最终数据库状态。

### 查询、筛选与分页

- 准备目标项与干扰项，验证包含、排除和准确顺序；
- 覆盖默认集合、单筛选、组合筛选、无效值和无结果；
- 验证稳定次级排序、页大小、总数以及翻页时筛选参数保留；
- 未发布、其他租户或越权数据不得泄漏。

### 队列、事件与通知

- 将“是否调度”与“任务执行结果”拆成两个合同；空任务也能通过入队断言；
- 断言事件/任务携带正确业务标识，不只断言类型；
- 通知验证接收者、渠道、载荷和不应收到的人；读取侧验证权限、分页与已读状态；
- 重试任务要验证幂等、失败恢复和旧任务不会覆盖新状态。

### 文件上传

- 使用 fake 文件和 fake 存储；
- 覆盖游客、越权、必填、真实类型、大小/尺寸边界和成功保存；
- 成功时同时断言业务记录与文件存在；失败时断言两者都未变化。

### 外部服务

- 快速测试用端口和确定性 Fake 验证应用行为；
- 适配器测试验证请求格式、响应映射、超时、错误和重试；
- 真实联网测试单独分组并在受控 CI 中运行，不进入高频红绿循环。

### 命令行

断言退出码、关键输出或错误信息，以及命令真正产生的数据库、缓存或文件结果。仅“没有抛异常”不算成功。

### 浏览器/E2E

- 只覆盖必须经过真实浏览器、客户端脚本、会话或前后端联动的高价值旅程；
- 使用独立环境和数据库，绝不连接开发或生产状态；
- 使用稳定的测试专用选择器，避免依赖样式层级、展示文案或易变布局；
- 等待可观察条件，不使用固定 `sleep`/`pause`；
- 失败时保留截图、控制台、网络和服务端日志；
- 不用 E2E 重复覆盖低层已能稳定证明的业务规则。

## 10. 扩大验证、证据与门禁

按反馈速度逐步扩大：

1. 当前测试；
2. 当前文件或能力模块；
3. 全部快速测试；
4. 必要的数据库、契约与适配器测试；
5. 高风险关键旅程的端到端测试。

推送前运行全部确定性快速测试。慢速、online 和浏览器套件放到独立 CI 阶段；本地 Git Hook 只能提供提醒，CI 才是不可绕过的团队门禁。

交付时保留：

- 行为与风险矩阵；
- 首次有效 RED 的命令、失败摘要和失败原因；
- 最终聚焦测试、相关套件及全量测试结果；
- 未运行测试及原因；
- Bug 对应的永久回归测试；
- 必要的浏览器截图、日志或外部契约证据。

## 11. 跨框架适配原则

只复用跨语言、跨框架的方法，不机械复制任何示例项目的版本、语法或目录：

- 不强制某一种测试标记、发现语法或命名风格，遵循目标项目约定；
- 按实际依赖边界归类测试，不依据示例文件位置或名称；
- 只在定位意外异常时临时暴露内部异常，最终恢复外部行为断言；
- 将固定时长等待改为可观察条件等待；
- 本地提交或推送钩子不能替代持续集成门禁；
- 适配目标项目当前支持的驱动、命令和目录结构；
- 对“先查询再创建”、常量 Fake 和仅验证任务调度等做法，补充并发、参数及最终行为验证。

## 常见失误

- 先完成实现再补测试，或修改测试期望只为让实现变绿；
- 把环境、夹具、过滤命令错误或类不存在当成最终 RED；
- 测试名描述实现而不是业务，或一个测试包含多个独立动作；
- 只断言状态码、数量、关系类型或内部调用；
- 只写成功路径，遗漏权限、反向状态、重复请求和禁止副作用；
- Fake 永远返回常量，真实网络进入快速测试；
- 只测试队列、事件或通知被调度，不测试最终行为；
- 用查询后创建假装解决并发，未设置唯一约束；
