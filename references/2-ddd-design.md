# DDD 设计模块

本模块用于把复杂业务知识变成可维护的边界、模型和协作契约。先做战略设计，再按实际复杂度选择战术模式；目录、类名和中间件都不是设计起点。

## 1. 适用性与投入门禁

DDD 的价值是管理业务认知和变化，不是增加对象数量。先判断投入级别：

| 观察到的事实 | 建议投入 |
| --- | --- |
| 规则很薄，主要是录入、查询和简单校验 | 轻量模块化，清楚分离输入、业务和持久化即可 |
| 有状态迁移、组合规则、不变量或权限语义 | 在对应模块使用实体、值对象和聚合 |
| 多团队对同一术语理解不同，模型所有权冲突 | 先做统一语言、子域和限界上下文 |
| 有跨上下文协作、外部模型污染或一致性难题 | 增加上下文映射、Port、防腐层、事件和流程管理 |
| 核心规则长期演进，错误代价高 | 对核心子域做完整战略与战术设计 |

不要因为存在数据库表、服务调用或多个目录就自动使用完整 DDD。简单 CRUD、一次性原型、纯技术转换和稳定的通用能力，优先选择更轻的方案。

新建上下文或聚合前先检查：

- 这是新的业务语言和规则所有权，还是现有模型的一个用例？
- 复杂度来自业务本身，还是来自技术实现、糟糕依赖或重复代码？
- 边界能否由团队解释并独立演进，而不只是为了让目录整齐？
- 使用模式后，变化点是否更容易定位、测试和替换？

## 2. 战略设计：语言、子域与限界上下文

### 2.1 建立统一语言

从用户对话、业务规则、现有代码、数据、测试和故障案例中提取名词、动作、状态、数量、时间与事件。术语必须同时进入讨论、文档、测试和代码。

维护最小术语表：

| 术语 | 精确定义 | 所属上下文 | 同义词 | 禁用或歧义用法 | 示例/反例 |
| --- | --- | --- | --- | --- | --- |

遇到以下信号时暂停建类，先澄清语言：

- 同一个词在不同角色口中含义不同；
- 不同词实际表达同一概念；
- 状态名称不能说明允许的行为；
- 技术词代替了业务意图；
- 一个布尔值隐藏了多个业务状态；
- “正常情况”“特殊处理”等词没有可验证定义。

不要追求全组织只有一套模型。统一语言在限界上下文内统一；跨上下文同名异义是允许的，但必须明确转换。

### 2.2 识别子域

按业务价值和差异化程度分类：

- **核心子域**：形成竞争差异、需要持续学习和演进的能力；投入最强建模力量。
- **支撑子域**：业务必需但不是主要差异来源；保持清晰边界，避免过度设计。
- **通用子域**：行业普遍存在且规则成熟；优先复用或采购，避免把核心精力耗在通用问题上。

分类会随战略变化而变化。记录“为什么这样分类”和复查触发条件，不把当前判断永久化。

子域是问题空间的业务分类，限界上下文是解法空间中的模型边界，两者不要求一一对应：一个上下文可以承载多个紧密相关的子域，一个复杂子域也可能由多个上下文协作实现。存量系统同时记录现状图、目标图和迁移步骤，不用理想边界覆盖真实依赖。

### 2.3 划定限界上下文

限界上下文是某套语言与模型成立的边界，不天然等同于部署单元、团队、代码仓库、数据库或网络服务。

用以下证据寻找边界：

- 术语定义、规则和状态机不同；
- 模型由不同角色或团队拥有；
- 变化节奏、合规要求或可用性目标不同；
- 同一实体需要不同的属性、历史或一致性；
- 一个能力可以通过稳定契约独立演进；
- 强行共享对象会让一方持续迁就另一方。

每个上下文写一张简报：

| 项目 | 内容 |
| --- | --- |
| 名称与业务目标 | 用业务语言说明它解决什么问题 |
| 核心语言 | 本上下文拥有的关键术语和定义 |
| 规则所有权 | 哪些不变量、政策和状态由它裁决 |
| 输入 | 接受的命令、查询或事件 |
| 输出 | 返回结果、公开事件或能力契约 |
| 数据所有权 | 哪些事实由它作为权威来源 |
| 不负责什么 | 明确排除项，防止边界膨胀 |
| 依赖 | 上下游及依赖原因 |
| 一致性与失败 | 强一致范围、延迟容忍和恢复方式 |

## 3. 上下文映射与协作关系

先画业务关系，再决定同步调用、事件或数据复制。为每条跨上下文关系记录：

| 上游 | 下游 | 关系模式 | 交换语义 | 契约所有者 | 一致性 | 失败与恢复 |
| --- | --- | --- | --- | --- | --- | --- |

可选关系模式：

- **合作关系**：双方共同规划并同步调整，适合目标一致且协作紧密的团队。
- **客户—供应者**：下游需求会影响上游优先级，双方明确版本和服务承诺。
- **遵奉者**：下游直接接受上游模型；只在转换成本高于耦合成本时使用，并记录代价。
- **防腐层**：下游通过翻译层保护自身语言，适合外部系统、遗留模型或语义差异明显的上下文。
- **开放主机服务/发布语言**：上游提供稳定、文档化、可版本化的公共协议。
- **共享内核**：双方共同拥有一小块模型；只共享真正稳定且必须一致的部分，变更需共同评审。
- **各行其道**：没有足够协作价值时保持独立，接受必要的重复。

上下文协作原则：

- 一个上下文不得直接修改另一个上下文的数据或内部对象；
- 不通过共享数据库表绕过公开契约；
- 跨上下文传递业务事实或稳定契约，不暴露内部对象图；
- 同步调用只表达当前步骤确实需要即时结果的依赖；
- 异步不等于无设计，仍需定义时序、重复、乱序、延迟和失败语义；
- 共享内核保持最小，不能成为“公共杂物箱”。

上述约束描述目标状态。若存量系统已经共享数据或跨边界调用，先把违规依赖、当前权威来源、影响范围、过渡契约、责任人和退出条件记录下来，再用接口包裹、变更捕获、双读校验或逐步替换迁移；不要假装旧耦合不存在，也不要要求一次性重写。

## 4. 四色建模

四色建模用于从业务事件和参与关系发现语言、边界与不变量。颜色只是视觉提示，文本中始终保留语义标签。

| 类型 | 识别问题 | 建模重点 |
| --- | --- | --- |
| 粉色 Moment-Interval | 发生了什么业务事实或持续过程？ | 时间、触发、前置条件、状态迁移、完成或失败事实 |
| 黄色 Role | 谁以什么身份参与？ | 权利、责任、权限与上下文身份；角色不等同于人或组织本身 |
| 绿色 Party/Place/Thing | 哪些有身份的参与者、地点或资源被涉及？ | 身份、所有权、生命周期，以及是否只是外部引用 |
| 蓝色 Description | 事实依据什么规格、政策、模板或目录？ | 版本、适用范围、快照时点，区分事实和对事实的描述 |

建模步骤：

1. 沿业务时间线列出粉色事件/过程，以及触发它们的命令。
2. 为每个事件补充黄色角色，标明谁发起、谁负责、谁受益或承担风险。
3. 连接绿色参与者、地点和资源，标明身份与所有权。
4. 连接蓝色规格、政策和模板，标明版本及历史快照要求。
5. 补充前置条件、状态迁移、数量/金额/时间约束、并发冲突和失败路径。
6. 按语言与规则所有权聚类，候选限界上下文由此浮现。
7. 再从身份、生命周期、值语义和原子不变量推导实体、值对象、聚合和事件。

验证问题：

- 每个事件是已经发生的事实，还是尚未表达意图的命令？
- 每个角色的权限依据什么规则，作用在哪个上下文？
- 描述对象改变后，历史事实是否应保持当时版本？
- 哪些规则必须同一原子操作成立，哪些允许延迟收敛？
- 重复、乱序、撤销、超时和人工介入时会发生什么？

不要把每张卡片机械生成一个类，也不要先画类图再倒填业务含义。

## 5. 战术建模与行为归属

### 5.1 Entity

Entity 由身份和连续生命周期定义。设计时明确：

- 身份在哪个范围内唯一、由谁生成、何时可用；
- 相等性按身份还是其他业务规则判断；
- 允许的状态迁移和终止条件；
- 哪些行为维护自身不变量；
- 并发修改如何检测。

Entity 通过有意图的行为改变状态，不公开可绕过规则的通用写入口。命令、请求参数、查询条件和传输对象不是 Entity。

### 5.2 Value Object

Value Object 由全部值共同定义，没有独立身份：

- 构造时验证，创建后不可变；
- 相等性按值判断；
- 封装单位、精度、范围、格式和相关运算；
- 修改通过返回新值表达；
- 不使用特殊空值或默认业务状态掩盖未知输入。

稳定且封闭的集合可建模为枚举；可能扩展或来自外部的值，必须显式处理未知情况，不能静默映射成某个正常状态。

### 5.3 Aggregate

Aggregate 是原子不变量和并发控制边界，不是对象数量、表关系或页面数据的集合。划分时追问：

> 哪些规则在一次业务提交结束时必须同时成立？

规则：

- Aggregate Root 是外部唯一修改入口；
- 内部集合只暴露只读视图或意图明确的查询；
- 聚合内维护强一致不变量；
- 聚合外只引用其他聚合的身份或必要快照；
- 聚合尽量小，只装入必须原子一致的对象；
- 使用版本或等价机制检测并发冲突；
- 聚合行为不执行网络、消息、文件或长耗时操作；
- 一个 Entity 也可以独立成为聚合；没有跨对象不变量时不要强行合并。

为每个聚合维护不变量表：

| 聚合 | 不变量 | 触发行为 | 修改入口 | 冲突策略 | 失败语义 |
| --- | --- | --- | --- | --- | --- |

### 5.4 Domain Service

只有当一项重要领域行为不自然属于某个 Entity 或 Value Object 时，才使用 Domain Service。它应：

- 用统一语言命名，表达计算、判断或领域动作；
- 无状态，依赖明确的领域能力；
- 不承担协议转换、事务、重试、消息发送、线程调度或通用编排；
- 不成为贫血对象的行为收容所。

跨聚合流程通常由应用用例编排；只有其中纯粹的业务决策才下沉为 Domain Service。

### 5.5 Domain Event

Domain Event 是领域内已经发生且业务关心的事实：

- 用过去时业务语言命名；
- 业务负载不可变；业务发生时间只有在属于领域事实时才进入负载；
- 在聚合完成合法状态迁移后记录；
- 只携带消费者理解事实所需的稳定数据；
- 需要持久化、发布或追踪时，由领域事件或外层事件信封提供事件身份、记录时间、聚合身份与版本、关联标识和因果标识。

区分两类事件：

- **领域事件**：服务于当前领域模型内的规则传播，可以保持较丰富的领域语义。
- **集成事件**：跨上下文公开的版本化契约，应稳定、最小并避免泄漏内部模型。

领域事件不是“任何状态变化日志”，也不能保证一经创建就已经可靠发布；可靠交付由应用与基础设施协作完成。

区分领域事件负载与持久化/集成事件信封。聚合不必为技术追踪生成随机身份或读取系统时钟；应用或可靠事件存储可以在边界补齐信封元数据。若时间本身参与业务判断，则通过明确的领域时间值传入，而不是在模型内部隐式读取。

### 5.6 Factory、Command 与 Query

- 创建过程需要多对象协作、复杂校验或不同创建策略时使用 Factory；简单构造不必包装。
- Command 表达调用者希望系统执行的用例意图，不是 Entity，也不应伪装成领域事实。
- Query 表达信息需求。复杂读场景可以直接使用专用读模型，不必为了查询加载和拼装聚合。

### 5.7 行为归属顺序

按最靠近规则、最小而完整的所有者放置行为：

1. 单个 Value Object 能完整表达：放 Value Object。
2. 单个 Entity 的身份与生命周期规则：放 Entity。
3. 同一原子边界内的不变量：放 Aggregate Root。
4. 无自然对象所有者的纯领域决策：放 Domain Service。
5. 加载、保存、跨聚合编排和副作用协调：放 Application Use Case。

如果行为放置需要访问大量外部细节，通常是边界或抽象层次有误。

## 6. 应用用例与六边形边界

概念依赖方向：

```text
Inbound Adapter → Input Port / Application Use Case → Domain
                                  ↓
                             Output Ports
                                  ↑
                           Outbound Adapters
```

这表示源码依赖指向更内层的业务策略，不规定进程、模块或目录数量。

职责分工：

- **Input Port**：由应用层公开一个用户目标的调用契约；它可以就是 Application Use Case 的接口。
- **Inbound Adapter**：依赖 Input Port，处理协议、提取认证身份、语法校验和输入/输出映射；不裁决业务规则。
- **Application Use Case**：代表一个用户目标，负责用例级访问控制、加载聚合、调用领域行为、保存结果、协调事务、幂等、事件和跨聚合步骤。
- **Domain**：表达状态、规则、不变量和决策；不感知传输、存储与运行框架。
- **Output Port**：由内层声明它实际需要的时钟、身份生成、持久化、支付、通知、消息或其他能力。
- **Outbound Adapter**：实现 Port，完成技术调用、数据映射和故障翻译。
- **Composition Root**：选择实现并完成依赖装配。

应用用例本身是概念责任，小型项目可以与其他代码共置，不必为了“分层”创建独立部署或空壳模块。

认证证明“调用者是谁”；应用级访问控制判断“是否允许调用该用例”；依赖领域状态、角色和不变量的授权规则由 Domain 裁决。三者不要全部塞进协议边界，也不要让 Domain 读取传输层身份对象。

错误处理原则：

- 领域错误保留业务语义，不带传输状态码；
- 外部故障在 Adapter 处翻译成 Port 契约允许的失败类型；
- Inbound Adapter 在单一边界统一映射公开响应；
- 不在多个层级重复捕获、吞掉或改写同一错误。

命令与查询可以共享模型，也可在复杂度有证据时分离。读模型优化不能反向污染写侧不变量。

## 7. Repository、Port、Adapter 与防腐层

### 7.1 Repository

Repository 是面向 Aggregate Root 的集合式抽象，也是输出 Port 的一种：

- 每个需要持久化的聚合按业务需要定义契约；
- 方法使用领域语言，而不是暴露通用数据操作；
- 返回聚合、领域值或显式的缺失结果；
- 保存时维持聚合版本和并发语义；
- 不泄漏存储记录、查询构造器、连接或缓存对象；
- 不用一个“万能基础仓储”抹平不同聚合的语义；
- 复杂报表和列表查询使用查询服务或读模型，不塞进聚合 Repository。

“不存在”“并发冲突”“存储不可用”是不同结果，不能统一成空值或布尔值。

### 7.2 Port

Port 是边界契约，不只用于远程调用：Input Port 表达外部可请求的应用用例，Output Port 表达内层为完成用例所需的能力。下列能力属于常见 Output Port：

- 获取当前时间或生成身份；
- 加载和保存聚合；
- 风险判断、计费、支付或库存承诺；
- 发送通知、发布消息或保存文件；
- 读取外部权威事实。

接口归最靠近且真正消费它的内层所有。若依赖是领域决策的一部分，可由 Domain 声明；若只是用例编排所需，由 Application 声明。不要建立与外部接口一比一复制、却没有业务语义的 Port。

### 7.3 Adapter 与防腐层

Adapter 实现 Port，并隔离技术与外部模型。防腐层尤其要转换：

- 外部身份与内部身份；
- 状态、枚举和生命周期语义；
- 金额、单位、精度、时区和日期；
- 数据版本与缺失字段；
- 外部错误到领域可理解的失败类型。

至少区分：事实不存在、业务拒绝、认证/授权失败、频率限制、超时、暂时不可用和不可恢复错误。不能把所有失败压成 `false`、空值或通用异常。

缓存是 Adapter 的实现细节，不改变 Repository/Port 的业务契约，也不能让 Domain 感知缓存命中。

## 8. 事务、一致性与长流程

### 8.1 本地事务

默认一个用例在一个本地事务中修改一个聚合。若同一事务修改多个聚合，必须给出不可分割的业务不变量证据，并评估锁竞争与并发冲突。

- 事务边界由应用用例控制；
- 事务内不执行外部网络调用或不可控长操作；
- 条件更新失败时区分数据不存在与版本冲突；
- 缓存写入/失效不与数据库提交假装成同一原子操作；
- 提交后失效、版本化缓存或对账机制要与可接受陈旧度匹配。

### 8.2 跨边界一致性

跨聚合或上下文优先使用显式业务流程与最终一致性。输出一致性矩阵：

| 步骤/事实 | 权威所有者 | 一致性要求 | 重复处理 | 乱序处理 | 超时/补偿 | 可观测证据 |
| --- | --- | --- | --- | --- | --- | --- |

可靠事件交付的通用闭环：

1. 聚合状态与待发布事件在同一本地事务持久化。
2. 独立发布器以至少一次语义转发事件。
3. 消费者用事件身份或业务幂等键去重，并让去重记录与业务状态变更处于同一本地事务；若不用去重记录，则业务写入本身必须有可证明的自然幂等或条件更新语义。
4. 记录重试、退避、积压、死信、回放和对账信息。
5. 需要顺序时按聚合身份与版本检测缺口、重复和乱序。

持久化或集成事件信封至少考虑：事件身份、类型、记录/发生时间、聚合身份与版本、契约版本、关联标识和因果标识。不要承诺基础设施无法提供的“绝不重复且绝不丢失”。

### 8.3 长流程与补偿

跨多个上下文、持续时间长且不能用单一事务完成时，使用显式流程管理器或 Saga：

- 记录当前步骤、已完成事实、下一动作和截止时间；
- 每个步骤和补偿都可重试、幂等、可观察；
- 补偿是新的业务动作，不是时间倒流或数据库回滚；
- 定义无法自动补偿时的人工介入与对账入口；
- 保留关联标识串联命令、事件和外部调用。

同步重试只用于已证明幂等的操作，并设置超时、退避和抖动；根据故障隔离需要考虑熔断与舱壁。不要在事务内休眠或无限重试。

## 9. 分包、命名与架构约束

优先按限界上下文或业务能力组织，再在上下文内部表达逻辑职责：

```text
contexts/
└── {business-context}/
    ├── domain/
    ├── application/
    ├── adapters/
    │   ├── inbound/
    │   └── outbound/
    └── composition/
```

这只是概念草图。适配目标项目已有习惯；逻辑边界可以落在同一目录或模块，只要依赖与所有权清晰。

约束：

- 依赖指向内层；Domain 不引用 Adapter 或运行框架；
- 跨上下文只通过公开契约、Port 或事件协作；
- 不跨边界导入另一个上下文的内部聚合、Repository 或存储模型；
- 公共包只放真正稳定、无争议的共享语义；
- 能用架构测试验证依赖规则时，将关键规则自动化；
- 不通过循环依赖、全局服务定位或静态访问绕过边界。

命名原则：

- 优先使用统一语言中的自然名词和动词，不强制技术后缀；
- Use Case 用动作和目标命名；
- Domain Event 用过去时事实命名；
- Policy、Specification、Factory 等名称只在对象确实承担相应语义时使用；
- Adapter 可带协议或技术名称，因为它负责技术边界；
- 不把目录名、注解、继承关系或固定后缀当作 DDD 合规证明。

## 10. 设计模式选择

先直接实现，再根据真实变化轴选择最小模式。行数、分支数量和类型数量只能提示复查，不能作为自动套模式阈值。

| 业务证据 | 候选方式 | 主要代价/检查 |
| --- | --- | --- |
| 规则短、稳定且线性 | 直接实现 | 保持意图清晰，不提前抽象 |
| 同一决策有可替换且独立演进的政策 | Strategy/Policy | 策略边界是否真会变化，选择逻辑放哪里 |
| 多条可组合、可解释的业务条件 | Specification | 避免把普通条件包装成对象海洋 |
| 状态决定允许行为和迁移 | 显式状态机 | 非法迁移、终态、并发与恢复是否完整 |
| 多个处理步骤按规则组合 | Pipeline/Chain | 顺序、短路、错误与上下文可见性 |
| 复杂规则需业务人员审阅 | 决策表/决策树/规则图 | 冲突、覆盖、版本和可测试性 |
| 算法骨架稳定，局部步骤变化 | Template Method 或组合 | 优先评估组合，避免继承锁死变化 |
| 合法创建需要多步协作 | Factory | 简单构造不要增加间接层 |
| 聚合持久化 | Repository | 契约保持领域语义，不退化成通用数据操作 |
| 外部模型可能污染内部语言 | Port + Adapter/ACL | 映射、错误语义和版本责任 |
| 事实需要跨边界传播 | Domain/Integration Event | 可靠交付、幂等、顺序与契约演进 |
| 长期跨上下文业务流程 | Process Manager/Saga | 状态持久化、补偿、超时与人工介入 |
| 读写模型差异有明确收益 | CQRS/专用读模型 | 数据延迟、双模型维护和排障成本 |

每个非平凡模式决策记录：

1. 观察到的变化点或复杂度证据；
2. 选择的模式及其责任边界；
3. 更简单方案为何不足；
4. 新增的间接层、运行成本和失败方式；
5. 何时应简化或移除该模式。

## 11. 建模流程与交付物

按以下顺序工作，并允许对前一步迭代：

1. 收集需求事实、角色、场景、规则、术语与未知项。
2. 建立统一语言，分类核心、支撑和通用子域。
3. 划分限界上下文，写上下文简报和 Context Map。
4. 用时间线、四色模型或事件工作坊梳理命令、事件、角色、规则和失败。
5. 识别 Entity、Value Object、Aggregate、Domain Service、Factory 和 Event。
6. 写聚合不变量、行为入口、事务边界和并发策略。
7. 为每个用例画出加载—决策—保存—副作用的最小序列。
8. 定义 Repository/Port 契约、Adapter 翻译和外部故障语义。
9. 选择跨边界一致性、幂等、重试、补偿、回放和观测方案。
10. 适配现有项目画包结构草图，只在有证据时选择设计模式。
11. 将模型追踪回需求与验收用例，列出假设、风险和待验证问题。

按范围输出必要产物，不为了完整而制造文档：

- 统一语言表与歧义清单；
- 子域分类与投入理由；
- 限界上下文简报和 Context Map；
- 四色时间线、命令、事件与角色；
- 聚合目录、不变量表和状态迁移；
- 关键用例序列；
- Repository/Port 与公开事件契约；
- 一致性、幂等、补偿和失败恢复矩阵；
- 分包与依赖草图；
- 模式/架构决策记录；
- 需求—上下文—聚合—用例的追踪关系；
- 未决问题、验证方式与决策负责人。

未知业务事实必须标注为未知或假设，不得凭建模经验臆造。

## 12. 设计门禁与常见反例

### 12.1 交付门禁

完成前检查：

- 每个术语在所属上下文内有稳定定义；
- 每条业务规则具有一个明确的权威治理来源；它可以是单个上下文，也可以是有共同评审机制的共享内核；入口可以重复防御性校验，但最终裁决语义不能分叉；
- 上下文边界有语言、能力或所有权证据；
- 每个聚合的不变量、修改入口、事务边界和冲突策略明确；
- 外部不能绕过 Aggregate Root 修改内部状态；
- Domain 不依赖协议、存储、消息、缓存或运行框架细节；
- Repository、Port 与 Adapter 的语义和所有者清楚；
- 跨上下文契约、版本、失败和一致性选择明确；
- 重复、乱序、超时、重试、补偿和人工恢复有答案；
- 设计模式由变化证据驱动，并记录了更简单方案；
- 设计可追踪到真实需求、场景和验收标准；
- 结构适配目标项目，没有机械复制示例技术栈或目录。

### 12.2 常见反例及修正

| 反例 | 问题 | 修正 |
| --- | --- | --- |
| 先按数据表生成领域对象 | 数据结构替代业务语言和行为 | 从用例、不变量与生命周期建模，再映射存储 |
| 所有属性都有公共写入口 | 任何调用者都能绕过规则 | 使用意图明确的行为并封装集合 |
| 值对象可变或未知值静默降级 | 历史语义漂移，错误被伪装成正常状态 | 构造校验、不可变、显式未知/失败 |
| 按关联关系把整张对象图放进一个聚合 | 事务膨胀、锁竞争、并发冲突 | 只保留必须原子一致的对象，其他用身份引用 |
| Command/请求对象被当成 Entity | 传输意图与领域身份混淆 | Command 留在应用边界，领域对象只表达业务概念 |
| Domain Service 承担所有业务 | 领域对象贫血，规则所有权不清 | 按 Value Object → Entity → Aggregate → Domain Service 放置 |
| Repository 等同“本地”，Port 等同“远程” | 技术位置替代语义 | 两者都是内层能力契约，Repository 专注聚合集合语义 |
| 通用 Repository 暴露任意 CRUD | 聚合边界和并发语义被绕开 | 为聚合按业务需要定义最小契约 |
| 跨上下文直接调用内部 Repository | 模型所有权和契约消失 | 使用公开用例、Port、ACL 或事件 |
| 在本地事务中调用外部系统 | 锁时间不可控，无法真正原子回滚 | 提交本地事实后用可靠消息与显式流程推进 |
| 发布消息与保存状态分开且无恢复 | 状态和事件可能永久不一致 | 同事务保存待发布事件，提供重试、去重、回放和对账 |
| 缓存写入被当成数据库事务的一部分 | 提交顺序和失败语义虚假 | 明确陈旧度，提交后失效或使用版本与对账 |
| 看到多个分支就套策略/责任链 | 间接层多于业务收益 | 按真实变化轴选模式，稳定规则直接实现 |
| 固定目录、后缀或注解被当成 DDD | 形式合规掩盖边界和规则问题 | 保持逻辑依赖与语义，物理结构适配项目 |
| 限界上下文一律拆成独立服务 | 把建模决策误当部署决策 | 先明确语言和所有权，再独立决定部署边界 |
| 异步被当成自动解耦 | 契约、时序和失败仍然耦合 | 明确事件语义、版本、幂等、顺序和恢复 |
| 把所有共享概念放进公共模块 | 形成无所有者的耦合中心 | 优先各自建模；共享内核保持最小并共同治理 |

DDD 交付的最终标准不是图和目录是否齐全，而是业务规则是否找到唯一、可测试、可演进的所有者，边界协作是否具有明确契约与失败语义。
