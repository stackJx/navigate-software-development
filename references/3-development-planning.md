# 开发思路模块

## 目标

把已确认需求转成可排序、可验证、可交付的开发切片。本模块是每次执行开发罗盘的第二个必选阶段，必须消费需求分析；命中 DDD 时同时吸收领域设计，不要求为了形成计划而强制使用 DDD。

## 1. 映射现有实现

先检查代码、测试、数据结构、接口和约定，建立：

```text
用例 → 现有模块/上下文 → 相关文件或符号 → 当前行为 → 缺口
```

不要只按用户故事猜文件。标记复用点、待修改点、新增点和受影响消费者。

## 2. 划分模块

若已有 DDD 产物，复用限界上下文。否则从用例集合、业务职责和信息所有权导出模块，不按页面、表或 MVC 目录拆分。

每个模块声明：

- 负责什么、不负责什么；
- 权威持有哪些信息和规则；
- 服务哪些用例；
- 对外提供什么契约；
- 依赖哪些能力及原因。

## 3. 建立依赖图

只有当 A 完成某用例必须使用 B 的能力或契约时，记录：

> A 依赖 B，因为……

排序步骤：

1. 建立“用例 → 模块 → 信息所有权”矩阵；
2. 建立能力级有向依赖边；
3. 检查循环；出现环时重审边界、拆能力或抽契约；
4. 拓扑分层：底层前置模块优先，同层可并行；
5. 提前验证支付、外部 API、迁移等高风险契约；
6. 权限、审计、可观测性随相关切片实现，不统一拖到最后。

避免用粗粒度模块名掩盖双向依赖。分别写清同步查询、异步事件、回调和结果通知。

## 4. 确定 MVP

对每项能力执行删除测试：

> 删除后，核心价值闭环、数据正确性、安全、兼容性或必要失败恢复是否仍成立？

分为“本次必须、可后置、明确不做”。MVP 是最小可验证纵向闭环，不是代码最少；不可删除权限、数据一致性和必要恢复。

## 5. 切成实施单元

每个切片必须端到端可观察，建议包含：

| 字段 | 内容 |
| --- | --- |
| Slice | 名称与对应 UC |
| 行为 | Given/When/Then |
| 代码范围 | 模块、文件、接口、数据变化 |
| 依赖 | 前置能力和外部契约 |
| 验证 | 自动测试或检查命令 |
| 风险 | 迁移、兼容、并发、回滚 |
| 完成定义 | 二元可判定结果 |

优先顺序：核心闭环 → 高风险未知 → 基础能力 → 扩展体验。一个切片太大时按业务状态或行为拆，不按 Controller/Service/Repository 技术层横切。

## 6. 编写开发顺序

输出每一步的目标、前置、修改范围、验证和完成门禁。明确可并行项，但不要让并行修改争用同一契约或数据迁移。

实现请求应给出首个可执行切片；方案请求到计划交付后停止。是否进入 TDD 由主路由决定。

## 门禁

- 每个范围内用例至少进入一个开发切片；
- 每条依赖有方向和原因，无未解释的环；
- 顺序符合依赖，MVP 能完成核心闭环；
- 每步有文件/模块范围、验证方式和完成定义；
- 数据迁移、兼容、回滚和横切风险已定位；
- 计划引用现有代码证据，不是泛化模板。

## 输出

在最终答复中使用固定 Markdown 二级标题 `## 开发思路`。依次输出：现状与缺口、模块职责、用例—模块矩阵、依赖图/层级、MVP 范围、纵向切片、实施顺序、并行项、风险和门禁结论。

产物按任务复杂度缩放：简单修复或小功能可以合并为短计划，但至少明确当前缺口、修改范围、实施步骤、验证方式和风险。即使用户只要求分析、不要求编码，也要输出建议的实施路径和停止点，不能只给需求结论。
