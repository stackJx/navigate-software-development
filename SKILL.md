# 开发罗盘

带固定骨架的阶段路由器。**需求分析**和**开发思路**每次必做且必须可见输出；**DDD** 和 **TDD** 由任务门禁决定是否加入。

## 固定执行合同

每次触发严格按此顺序执行，只能跳过可选阶段：

```text
① 需求分析（必选，输出 ## 需求分析）
        ↓
② DDD（门禁命中时；否则记跳过理由）
        ↓
③ 开发思路（必选，输出 ## 开发思路，消费①与②）
        ↓
④ TDD（门禁命中时；否则记跳过理由，消费①的验收用例与③的切片）
        ↓
⑤ 实现与验证（仅当用户要求实现）
```

1. 读取 [1-requirements-analysis.md](references/1-requirements-analysis.md) 完成①，在答复中输出独立二级标题 `## 需求分析`。
2. 检查 DDD 门禁；命中读取 [2-ddd-design.md](references/2-ddd-design.md)，未命中记录跳过理由。
3. 读取 [3-development-planning.md](references/3-development-planning.md) 完成③，输出独立二级标题 `## 开发思路`。
4. 检查 TDD 门禁；命中读取 [4-tdd.md](references/4-tdd.md)，未命中记录跳过理由。
5. 用户要求实现时继续编码与验证；只要求分析或方案时，在③交付后停止。

两个必选标题不得改成加粗、路由说明或其他名称，内容也不得省略或只留在内部推理。产物可按复杂度缩放；已有可信上游产物可复用，但最终答复仍须概括本次采用的需求结论与开发方案。不确定输出形态时，对照 [5-worked-example.md](references/5-worked-example.md)(含跳过路径与全命中路径两个示例)。

## 路由表

| 用户意图 | 执行顺序 | 默认停止点 |
| --- | --- | --- |
| 需求分析、范围、角色、用例、验收标准 | ①→③ | 输出需求分析与开发思路，不擅自编码 |
| 开发思路、开发方案、模块梳理、优先级、MVP | ①→③ | 输出需求分析与可执行开发计划 |
| DDD、领域建模、上下文映射、聚合、领域事件、六边形架构、领域模式 | ①→②→③ | 输出需求分析、领域设计与开发思路 |
| TDD、测试先行、测试命名/设计、测试替身、回归测试 | ①→②(命中时)→③→④ | 输出需求分析、开发思路及测试合同/结果 |
| 完整功能开发或重构 | ①→②(命中时)→③→④(命中时)→⑤ | 完成实现与验证，并保留两个必选输出 |
| Bug 修复 | ①→②(命中时)→③→④(命中时)→⑤ | 修复与验证，并保留两个必选输出 |

## DDD 选择门禁

满足任一条件时加入 DDD 模块：

- 用户明确要求 DDD、四色建模、聚合、分包或领域模式；
- 存在丰富业务规则、状态迁移、不变量或长期演进的统一语言；
- 存在多业务能力、模型所有权冲突、跨模块一致性或外部系统防腐需求；
- 问题本质是上下文边界、聚合边界、依赖方向、职责放置或跨边界一致性错误。

简单 CRUD、一次性原型、纯技术适配或规则很薄时跳过 DDD 并简述理由。跳过 DDD 不等于跳过模块边界与依赖检查。命中时也可只做与问题相关的**局部**战术设计，不必每次铺满全套战略建模（见 [5-worked-example.md](references/5-worked-example.md) 示例二）。

## TDD 选择门禁

满足任一条件时加入 TDD 模块：

- 用户明确要求 TDD、测试先行或回归测试；
- 用户要求设计或改进测试命名、结构、分层、夹具、替身或可读性；
- 正在实现或修复业务规则、权限、状态、边界值、幂等、并发或副作用；
- 代码库已有可运行测试，变更需要回归保护；
- Bug 可先用稳定失败用例复现。

纯需求分析、纯开发计划、探索性原型或非可执行文档不进入红—绿—重构。若用户明确不要测试，遵从并报告未覆盖风险。

## 子智能体委派（可选执行优化）

这是 **Codex 执行层的可选优化，不是第 6 个方法阶段**：把某阶段内独立、无共享状态的子任务 fan-out 给子智能体并行推进，再由主智能体扇入合并。方法骨架、门禁逻辑与两个必选章节的固定输出一律不变。此处「智能体并行」指用多个子代理同时干活，与开发思路里「同层模块/代码切片可并行」是两码事，勿混淆。

前置：需在 `~/.codex/config.toml` 开启 `[features] multi_agent = true`，启用工具 `spawn_agent` / `wait_agent` / `close_agent`。

### 委派门禁（何时 fan-out）

满足**全部**条件才委派：

- 当前阶段能切出 ≥3 个边界清晰、无共享状态、可自包含描述的子任务；
- 并行收益明显大于扇入合并成本；
- 子任务产物能各自独立收敛，不需要彼此的中间结果。

简单任务、小修复、强耦合、需要全局共享上下文、或实现类会互相争用同一契约/迁移时**不委派**——允许结论就是「不委派、主智能体串行完成」（对齐「设计模式由变化点驱动，允许不使用模式」）。委派与否按复杂度缩放,简单 CRUD 不 spawn,如同简单任务跳过 DDD。

### 检测与降级（fan-out 前必查）

依次检查，任一不满足即降级为 solo（主智能体自己串行做）：

1. **递归守卫**：当前若已在子代理内执行，不再递归 spawn，直接 solo；
2. **能力可用**：`multi_agent` 未开启或 `spawn_agent` 工具不可用 → solo；
3. **运行时失败**：spawn 返回失败、`agent_type` 不可用或子代理无法启动 → 停止继续 spawn，改 solo，不把部分结果当完整结论。

无论委派还是降级,都在答复中一句话说明**实际执行方式与原因**（如「已并行委派 4 个需求角色梳理」或「multi_agent 未开启，降级 solo」）。skill 在未开启 `multi_agent` 的环境下默认走 solo，开箱即用。

### 各阶段拆分建议

| 阶段 | 可并行 fan-out | 必须由主智能体集中/串行 |
| --- | --- | --- |
| ① 需求分析 | 按角色/用例并行梳理 | 合并术语表、缝合跨角色交接链、消歧 |
| ② DDD | 每聚合/每上下文的局部战术设计 | 统一语言、Context Map、跨边界一致性矩阵 |
| ③ 开发思路 | **依赖图建立后**按同层模块/纵向切片 | 依赖图构建；避免并行切片争用同一契约或迁移 |
| ④ TDD | 独立切片/不同测试层并行 | 先建绿灯基线；外向内推进（暂停外层→窄测试→回外层）串行不可拆；证据汇总 |
| ⑤ 实现验证 | 独立切片并行编码（同③约束） | 扩大验证收敛 |

### 调用与扇入

- 调用：`spawn_agent({ task_name, fork_turns: "none", message })` —— `fork_turns: "none"` 表示全新隔离上下文，故 `message` 必须**自包含**（子代理不继承会话历史，需带齐路径、范围、约束与期望产出格式）；`wait_agent({ timeout_ms })` 收集结果；`close_agent({ target })` 释放槽位。
- 约定：**分析/建模/审查类可并行；实现类默认串行**避免文件冲突。可复用随附模板 [agents/navigator-worker.toml](agents/navigator-worker.toml)（复制到 `~/.codex/agents/` 或项目 `.codex/agents/` 后按 `agent_type` 引用）。
- **扇入合并（reconverge）**：所有 fan-out 结果**必须**并回 `## 跨阶段追踪` 的单一链条与两个固定必选章节；子产物视同「上游产物」，最终答复须**摘要采用结论**，不留多份平行产物、不臆造规则；阻塞性业务决策仍由主智能体显式提给用户，不替用户拍板。

## 分发与执行规则

1. 读取用户请求、现有代码、文档与测试，区分事实、推断与未知项；先检查现有仓库约定，再提出结构或改代码。
2. 始终执行①和③，只对②和④应用门禁；上游产物可信时复用，不重复调查，但要摘要与本次相关的结论。
3. 业务事实必须有用户输入、现有代码、测试或文档证据，不臆造规则；非阻塞未知项可采用明确、可逆的安全假设，阻塞性业务决策再询问用户。
4. 优先交付最小纵向闭环；权限、数据正确性与必要失败恢复不能以 MVP 为由删除。
5. 设计模式由变化点或复杂度证据驱动，允许结论是"不使用模式"；适配目标项目的语言、框架与目录习惯，不机械复制任何示例结构。
6. 保留并尊重现有测试，不能为绿灯弱化断言、吞异常或跳过测试。

## 跨阶段追踪

为必选阶段与实际选中的可选阶段维护最小追踪关系：

```text
需求/用例 → 模块或上下文（若有） → 开发切片 → 测试（若有） → 代码 → 验证证据
```

不得留下无对应需求的设计，也不得把范围内用例遗漏在计划之外。需求分析与开发思路不得标注"未选"；未选 DDD 或 TDD 时在对应位置标注"未选及理由"，不伪造产物。

## 交付格式

最终答复按此顺序；第 1、3 项是不可省略的可见产物：

1. **需求分析（必选）**：核心目标、范围、关键用例/验收标准、约束与未知项；复杂任务按模块 1 完整展开。
2. **领域设计（可选）**：命中 DDD 时输出设计合同；否则一句话给出跳过理由。
3. **开发思路（必选）**：现状/缺口、实现范围、纵向切片或步骤、依赖、验证方式与风险；复杂任务按模块 3 完整展开。
4. **TDD（可选）**：命中门禁时输出测试合同、循环或测试证据；否则一句话给出跳过理由。
5. **实施与验证**：用户要求实现时，汇报修改文件、关键决策、执行检查及结果。
6. **未决风险**：只列真实未解决项与需用户决定的事项。

即使已完成编码，也要保留 `## 需求分析` 和 `## 开发思路` 章节说明实际实现依据，不得只汇报文件与测试结果。

## 资料基础

本 Skill 综合提炼需求分析、领域设计、开发规划与测试驱动开发资料。四色建模作为连接需求分析与领域设计的标准建模步骤。[5-worked-example.md](references/5-worked-example.md) 提供技术栈中立的端到端走查,用于校准输出形态。子智能体委派的预检降级与并行 fan-out 范式,借鉴 Codex 生态中 story-review 与 maestro-flow 的实践。
