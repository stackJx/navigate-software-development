## 分发规则

1. 读取用户请求、现有代码、文档和测试，区分事实、推断与未知项。
2. 选择满足当前请求的最小模块集合。不要读取未选模块。
3. 上游产物已存在且可信时直接复用，不重复分析。
4. 缺少下游必需输入时，只补最小上游：
   - DDD 需要清晰业务语言和规则；缺失时补需求分析。
   - 开发计划需要范围与验收用例；缺失时补需求分析。
   - TDD 需要可观察行为；缺失时补需求分析，不必自动补 DDD。
5. 用户只要求分析或方案时，到对应停止点交付，不擅自编码。
6. 用户要求实现时，完成选定阶段后继续修改代码并验证；只有阻塞性业务决策才暂停询问。

## DDD 选择门禁

满足任一条件时选择 DDD 模块：

- 用户明确要求 DDD、四色建模、聚合、分包或领域模式；
- 存在丰富业务规则、状态迁移、不变量或长期演进的统一语言；
- 存在多个业务能力、模型所有权冲突、跨模块一致性或外部系统防腐需求；
- 当前问题本质是上下文边界、聚合边界、依赖方向、职责放置或跨边界一致性错误。

简单 CRUD、一次性原型、纯技术适配或规则很薄时跳过 DDD，并简述理由。跳过 DDD 不等于跳过模块边界和依赖检查。

## TDD 选择门禁

满足任一条件时选择 TDD 模块：

- 用户明确要求 TDD、测试先行或回归测试；
- 用户要求设计或改进测试命名、结构、分层、夹具、替身或测试可读性；
- 正在实现或修复业务规则、权限、状态、边界值、幂等、并发或副作用；
- 代码库已有可运行测试，变更需要回归保护；
- Bug 可以先用稳定失败用例复现。

纯需求分析、纯开发计划、探索性原型或非可执行文档不进入红—绿—重构。若用户明确不要测试，遵从并报告未覆盖风险。

## 阶段组合

选中多个模块时，按以下依赖顺序执行，但跳过未选阶段：

```text
需求分析 ──┬──> 开发思路 ──> TDD（实现时可选）
           └──> DDD ────────> 开发思路
```

开发思路可以直接消费需求分析产物，不要求先做 DDD。TDD 可以直接消费验收用例和开发切片，不要求先做 DDD。

## 跨阶段追踪

为实际选择的阶段维护最小追踪关系：

```text
需求/用例 → 模块或上下文（若有） → 开发切片 → 测试（若有） → 代码 → 验证证据
```

不得留下无对应需求的设计，也不得把范围内用例遗漏在计划之外。未选择 DDD 或 TDD 时，在对应列标注“未选及理由”，不要伪造产物。

## 通用执行规则

- 先检查现有仓库约定和用户改动，再提出结构或修改代码。
- 业务事实必须有用户输入、现有代码、测试或文档证据；不要臆造业务规则。
- 非阻塞未知项可采用明确、可逆的安全假设；阻塞性决策再询问用户。
- 优先交付最小纵向闭环，权限、数据正确性和必要失败恢复不能以 MVP 为由删除。
- 设计模式必须由变化点或复杂度证据驱动；允许结论是“不使用模式”。
- 适配目标项目的语言、框架和目录习惯，不机械复制任何示例项目的结构。
- 保留并尊重现有测试；不能为了绿灯弱化断言、吞异常或跳过测试。

## 交付格式

按实际选中的模块汇报：

1. 本次路由：选中了哪些模块，为什么；跳过了哪些阶段。
2. 阶段产物：使用对应参考文件规定的格式。
3. 实施结果：修改文件、关键决策及兼容性影响。
4. 验证证据：执行的检查或测试及结果。
5. 未决风险：只列真实未解决项和需要用户决定的事项。
