# 开发罗盘

把本 Skill 当作带固定骨架的阶段路由器。需求分析和开发思路是每次执行的必选阶段；DDD 和 TDD 才由任务门禁决定是否加入。

## 固定执行合同

每次触发本 Skill，严格按以下顺序执行：

1. 读取 [1-requirements-analysis.md](references/1-requirements-analysis.md)，完成需求分析，并在最终答复中输出独立的 Markdown 二级标题 `## 需求分析`。
2. 检查 DDD 门禁；命中时读取 [2-ddd-design.md](references/2-ddd-design.md)，未命中时记录跳过理由。
3. 读取 [3-development-planning.md](references/3-development-planning.md)，消费需求分析及可选 DDD 产物，并在最终答复中输出独立的 Markdown 二级标题 `## 开发思路`。
4. 检查 TDD 门禁；命中时读取 [4-tdd.md](references/4-tdd.md)，未命中时记录跳过理由。
5. 用户要求实现时继续编码和验证；只要求分析或方案时，在开发思路交付后停止。

需求分析和开发思路可以按任务复杂度缩放，但固定标题不得改成加粗文本、路由说明或其他名称，内容也不得省略或只留在内部推理。已有可信上游产物可以复用，但最终答复仍须概括本次采用的需求结论和开发方案。

## 分发规则

1. 读取用户请求、现有代码、文档和测试，区分事实、推断与未知项。
2. 始终执行模块 1 和模块 3；只为模块 2 和模块 4 应用选择门禁。
3. 上游产物已存在且可信时直接复用，不重复调查，但要输出与本次任务相关的结论摘要。
4. DDD 需要清晰业务语言和规则，使用模块 1 的结果作为输入；信息不足时在需求分析中明确未知项。
5. TDD 需要可观察行为，使用模块 1 的验收用例和模块 3 的开发切片作为输入，不必自动补 DDD。
6. 用户只要求分析或方案时，在输出需求分析和开发思路后停止，不擅自编码。
7. 用户要求实现时，完成两个必选阶段及命中的可选阶段后继续修改代码并验证；只有阻塞性业务决策才暂停询问。

## DDD 选择门禁

满足任一条件时选择 DDD 模块：

- 用户明确要求 DDD、四色建模、聚合、分包或领域模式；
- 存在丰富业务规则、状态迁移、不变量或长期演进的统一语言；
- 存在多个业务能力、模型所有权冲突、跨模块一致性或外部系统防腐需求；
- 当前问题本质是上下文边界、聚合边界、依赖方向、职责放置或跨边界一致性错误。

简单 CRUD、一次性原型、纯技术适配或规则很薄时跳过 DDD，并简述理由。跳过 DDD 不等于跳过模块边界和依赖检查。

## TDD 选择门禁

满足任一条件时选择 TDD 模块：

- 用户明确要求 TDD、测试先行或回归测试；
- 用户要求设计或改进测试命名、结构、分层、夹具、替身或测试可读性；
- 正在实现或修复业务规则、权限、状态、边界值、幂等、并发或副作用；
- 代码库已有可运行测试，变更需要回归保护；
- Bug 可以先用稳定失败用例复现。

纯需求分析、纯开发计划、探索性原型或非可执行文档不进入红—绿—重构。若用户明确不要测试，遵从并报告未覆盖风险。

## 阶段组合

按以下固定骨架执行，只能跳过可选阶段：

```text
需求分析（必选并输出）
        ↓
DDD（门禁命中时）
        ↓
开发思路（必选并输出）
        ↓
TDD（门禁命中时）
        ↓
实现与验证（用户要求实现时）
```

开发思路必须消费需求分析产物；命中 DDD 时也要吸收领域设计决策。TDD 消费验收用例和开发切片，不要求先做 DDD。

## 跨阶段追踪

为必选阶段和实际选中的可选阶段维护最小追踪关系：

```text
需求/用例 → 模块或上下文（若有） → 开发切片 → 测试（若有） → 代码 → 验证证据
```

不得留下无对应需求的设计，也不得把范围内用例遗漏在计划之外。需求分析和开发思路不得标注为“未选”；未选择 DDD 或 TDD 时，在对应列标注“未选及理由”，不要伪造产物。

## 通用执行规则

- 先检查现有仓库约定和用户改动，再提出结构或修改代码。
- 业务事实必须有用户输入、现有代码、测试或文档证据；不要臆造业务规则。
- 非阻塞未知项可采用明确、可逆的安全假设；阻塞性决策再询问用户。
- 优先交付最小纵向闭环，权限、数据正确性和必要失败恢复不能以 MVP 为由删除。
- 设计模式必须由变化点或复杂度证据驱动；允许结论是“不使用模式”。
- 适配目标项目的语言、框架和目录习惯，不机械复制任何示例项目的结构。
- 保留并尊重现有测试；不能为了绿灯弱化断言、吞异常或跳过测试。

## 交付格式

最终答复使用以下顺序；第 1、3 项是不可省略的可见产物：

1. **需求分析（必选）**：至少输出核心目标、范围、关键用例/验收标准、约束与未知项；复杂任务按模块 1 的完整格式展开。
2. **领域设计（可选）**：命中 DDD 门禁时输出设计合同；否则用一句话给出跳过理由。
3. **开发思路（必选）**：至少输出现状/缺口、实现范围、纵向切片或步骤、依赖、验证方式和风险；复杂任务按模块 3 的完整格式展开。
4. **TDD（可选）**：命中门禁时输出测试合同、循环或测试证据；否则用一句话给出跳过理由。
5. **实施与验证**：用户要求实现时，汇报修改文件、关键决策、执行检查及结果。
6. **未决风险**：只列真实未解决项和需要用户决定的事项。

即使已经完成编码，也要保留 `## 需求分析` 和 `## 开发思路` 章节，说明实际实现依据；不得只汇报文件和测试结果。
